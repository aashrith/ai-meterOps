# C4 Diagrams

These diagrams show the service from two angles: the external context and the internal container split.

## Context View

```mermaid
C4Context
title AI Usage Metering and Quota Service - Context

Person(client, "API client", "Submits prompts and inspects usage")
System(service, "AI usage metering service", "Enforces quota and records billable AI usage")

Rel(client, service, "HTTPS / JSON")
```

## Container View

```mermaid
C4Container
title AI Usage Metering and Quota Service - Container

Person(client, "API client", "Submits prompts and inspects usage")

System_Boundary(service, "AI usage metering service") {
    Container(api, "FastAPI API", "HTTP routes, request validation, auth hooks", "FastAPI")
    Container(app, "Application services", "Use cases, quota orchestration, reconciliation", "Python")
    Container(ai, "AI provider adapter", "Mocked or real generation backend", "Pluggable port")
    ContainerDb(db, "Postgres", "User config, reservations, usage ledger", "PostgreSQL")
}

Rel(client, api, "HTTPS / JSON")
Rel(api, app, "Calls use cases")
Rel(app, ai, "Generates text")
Rel(app, db, "Reads and writes state")
Rel(ai, db, "No direct access")
```

## Reading the diagrams

- The context view is the “what talks to what” picture.
- The container view is the “how the service is split” picture.
- The FastAPI layer stays thin.
- The application layer owns the quota and credit rules.
- Postgres is the durable source of truth.
- The AI provider remains pluggable so we can swap mock and real backends without changing the domain.
